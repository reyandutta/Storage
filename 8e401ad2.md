# Security Assessment of Alibaba Cloud PAI‑Lingjun Intelligent Computing Service

## Executive summary

PAI‑Lingjun (Lingjun) is Alibaba Cloud’s high‑performance AI computing service that provides large‑scale heterogeneous GPU clusters, high‑bandwidth RDMA networking, and parallel storage for foundation model training and other HPC workloads.[^1][^2][^3][^4] From a security perspective, Lingjun relies heavily on core Alibaba Cloud controls—VPC networking, KMS‑backed encryption, RAM‑based IAM, and centralized logging—so the overall risk posture depends on how these shared controls are designed and operated.  
Lingjun can be deployed as PAI resource groups and also as ACK Lingjun Kubernetes clusters, which must be placed in VPCs with correctly segmented CIDR blocks and associated security groups.[^3][^5][^6] By combining disk and storage encryption, end‑to‑end TLS, least‑privilege RAM policies, and comprehensive audit logging via ActionTrail and Simple Log Service (SLS), tenants can bring Lingjun environments in line with stringent regulatory and internal security requirements.[^7][^8][^9][^10][^11][^12]

## 1. Service overview, uses, and setup

### 1.1 Service description and architecture

PAI‑Lingjun is a PaaS offering that delivers large‑scale heterogeneous computing power for high‑performance AI training and HPC workloads, eliminating the complexity of building and operating physical GPU clusters and RDMA networks.[^1][^2] The service is built on Panjiu GPU servers connected by high‑speed RDMA networks (up to 800 Gbit/s) and backed by the Cloud Parallel File System (CPFS) for large‑throughput storage.[^1][^2][^3][^4] Lingjun clusters organize compute into clusters, node groups, and nodes, where nodes are high‑performance GPU servers running supported OS images such as CentOS 7.9 and instrumented with Lingjun’s optimization suite.[^3]

### 1.2 Primary use cases

Lingjun targets large‑scale AI scenarios such as computer vision, NLP, recommendation systems, and training/deploying large language models, and it is marketed for industries including autonomous driving, finance, pharmaceutical research, scientific computing, and enterprise SaaS.[^1][^2][^13] Alibaba Cloud positions Lingjun as the backbone of large AI “super‑factory” clusters, achieving very high GPU utilization through orchestration components such as HPN 7.0 networking, an AI master scheduler, and CPFS integration.[^4][^14] In many environments Lingjun is consumed together with PAI workspaces, EAS (Elastic Algorithm Service), ACK (Kubernetes), and data services such as OSS, NAS, CPFS, and ApsaraDB, which must be included in any end‑to‑end security design.[^2][^3][^6][^15]

### 1.3 Deployment and setup patterns

Administrators create Lingjun resource groups and resource pools in PAI and then carve out resource quotas that are bound to workspaces for development and online inference workloads.[^16][^5] PAI allows you to define multiple Lingjun clusters and group nodes with similar specs into node groups, enabling efficient resource placement and scheduling.[^3][^5] For containerized workloads, ACK provides an ACK Lingjun managed‑cluster type that runs in a VPC, requires non‑overlapping private CIDR blocks (for VPC, service CIDR, and Lingjun interconnect ranges), and integrates with security groups, SLB, and Auto Scaling.[^6]

## 2. Data protection and encryption

### 2.1 Data at rest

Lingjun itself is a compute platform and persists data primarily through underlying Alibaba Cloud storage services such as ECS disks, CPFS, OSS, NAS, and database services, all of which support encryption at rest via KMS.[^2][^8][^9][^17] ECS disk encryption uses a two‑tier key hierarchy where a data key encrypts disk contents and is itself encrypted by a KMS key that resides in Key Management Service, with service‑managed keys (alias/acs/ecs) or customer‑managed keys available.[^9] The disk encryption process is transparent to instances, and encryption is irreversible—encrypted disks cannot be reverted to plaintext—which is relevant for lifecycle planning and incident response.[^9]

OSS supports both client‑side and server‑side encryption so objects written from Lingjun workloads can be encrypted either before transmission or by OSS using managed keys.[^8] For relational data used in AI workloads (for example, feature stores and label data), RDS provides Transparent Data Encryption (TDE) as well as SSL/TLS, and other services such as NAS and CPFS provide durable, replicated storage that can be combined with KMS‑based encryption and strict access controls.[^7][^8][^2][^15] KMS offers default keys for many cloud services plus software and hardware customer master keys; default keys can be used for basic cloud‑service encryption but are not suitable for application‑level encryption where explicit key separation and rotation policies are required.[^7]

### 2.2 Data in transit

Alibaba Cloud recommends encrypting all network transmission of sensitive business data using HTTPS and SSL/TLS, typically via SSL Certificate Service to terminate certificates on load balancers or application endpoints.[^7] SSL Certificate Service simplifies obtaining and deploying certificates on Alibaba Cloud resources so Lingjun‑fronting services (APIs, dashboards, inference endpoints) can be upgraded from HTTP to HTTPS with identity authentication and confidentiality for clients.[^7] For database connections and other backend services, Alibaba Cloud services such as RDS and some storage offerings support SSL/TLS transport, and KMS can be integrated at the application layer to encrypt data before it traverses networks.[^7][^8]

Lingjun’s internal networks use high‑bandwidth RDMA fabrics (Fat‑Tree topology with 25 Gbps management and 100 Gbps+ training networks), but this focuses on performance and high availability rather than explicit encryption, so tenants should assume that confidentiality is enforced via VPC isolation and upper‑layer TLS, not RDMA link‑level encryption.[^1][^2][^3] ACK Lingjun clusters must be deployed into VPCs with private address ranges, and all access from the public Internet should be mediated through SLB/ALB with HTTPS listeners and optional WAF, ensuring a TLS‑terminated edge for user‑facing workloads.[^6][^17]

### 2.3 Key management

Alibaba Cloud Key Management Service is the central service for managing keys for disk encryption, OSS server‑side encryption, and application‑level cryptography in Lingjun environments.[^7][^8][^9] KMS supports default keys (Aliyun_AES_256) that are tightly integrated with specific cloud services and customer‑managed software or hardware keys that can be scoped to particular applications or business domains.[^7] KMS keys benefit from centralized lifecycle management, including creation, rotation, disabling, and access control via RAM policies, and logs of key usage can be fed into SLS for audit purposes.[^7][^17][^11]

## 3. Infrastructure security

### 3.1 Compute, network, and storage architecture

Lingjun clusters comprise high‑performance heterogeneous compute nodes (Panjiu GPU servers) interconnected by an RDMA network and backed by distributed storage (CPFS), giving the appearance of a single large AI super‑cluster for tenant workloads.[^1][^2][^3][^4] Clusters are divided into node groups, which pool nodes with similar specs, and nodes themselves run vendor‑provided OS images plus Lingjun optimization components for data loading, collective communication, and resource optimization, which must be treated as privileged software from a hardening perspective.[^3]

ACK Lingjun managed clusters run inside Alibaba Cloud VPCs, and cluster creation enforces private CIDR allocations for the VPC, service CIDR, Lingjun interconnect CIDR, and Lingjun CIDR to avoid conflicts, ensuring that tenant workloads are isolated in private address spaces.[^6] Standard Alibaba Cloud network controls—security groups, route tables, SLB/ALB, and VPC peering—are used to expose or restrict Lingjun workloads, and quotas exist for the number of security groups, SLB instances, and EIPs per account, which have operational implications for large multi‑tenant environments.[^6]

### 3.2 Shared responsibility and platform controls

Alibaba Cloud provides platform‑level security controls for physical infrastructure, datacenter security, basic network isolation, and many default encryption features across storage and compute services.[^17][^9][^8] However, configuration of VPC boundaries, security groups, SLB listeners, WAF policies, and Security Center posture management remains the responsibility of the tenant, including for Lingjun and ACK Lingjun clusters.[^17][^6] Tenants must therefore design network segmentation (for example, separating training, inference, and management planes) and enforce least‑privilege access to Lingjun management APIs and consoles.

## 4. Identity and access management (IAM)

### 4.1 RAM integration with PAI and Lingjun

Platform for AI is fully integrated with Alibaba’s Resource Access Management (RAM) service, and all operations on PAI resources—including Lingjun workloads associated with workspaces—are authenticated and authorized by RAM.[^18][^19] RAM policies in PAI can make use of condition keys such as pai:Accessibility (resource visibility), pai:EntityAccessType (creator vs non‑creator), and acs:SourceIp (caller IP), enabling fine‑grained control over who can access Lingjun resources and from where.[^18] PAI also organizes compute resources into resource pools and hierarchical resource quotas, and custom RAM policies can limit access to specific resource pools or quotas identified by ARNs (for example, acs:pai:*:*:resourcegroup/{name}).[^16]

### 4.2 Accounts, roles, and permissions

RAM is a central identity and access management service that lets you create RAM users, groups, and roles, manage their credentials (passwords, AccessKeys, MFA), and attach JSON‑based policies to control access to Alibaba Cloud services.[^20][^19] Best practice guidance strongly recommends avoiding day‑to‑day use of the root Alibaba Cloud account; instead, administrators should create individual RAM users and use roles (via STS) for programmatic access, while enabling MFA and refraining from creating AccessKeys for the root account.[^20][^21] When managing Lingjun resource pools and clusters, Alibaba Cloud notes that the AliyunPAIFullAccess system policy gives full control over all PAI resources and must be granted cautiously, preferring more limited custom policies wherever possible.[^5]

RAM and related security advisories further recommend strong password policies (minimum length, character diversity), password retry constraints, enforced password rotation, disabling console access or AccessKeys for inactive users, and avoiding wildcard "*:*" administrative policies.[^21][^22][^23] Security Center best‑practice guidance also emphasizes granting permissions via groups, restricting source IP addresses in policies, rotating AccessKeys, and limiting full‑privilege policies, all of which directly apply to personnel managing Lingjun clusters and workloads.[^21][^22][^24]

## 5. Logging, monitoring, and audit

### 5.1 Metrics and health monitoring

CloudMonitor provides metrics‑based monitoring for many Alibaba Cloud services, and integrates with SLS to visualize metrics, set thresholds, and trigger alerts; the legacy log monitoring feature was retired in 2024 in favor of SLS‑based log monitoring in Hybrid Cloud Monitoring.[^25][^26] For SLS itself, CloudMonitor exposes metrics such as write traffic, raw data size, QPS, operation counts, and error counts, allowing you to monitor log ingestion and query health that underpin Lingjun observability pipelines.[^26]

### 5.2 Lingjun and PAI service logs

PAI supports configuring log collection for resource groups (for example, EAS resource groups that may be backed by Lingjun compute), where service logs are shipped to SLS projects and logstores that you specify.[^27] For dedicated resource groups, you can enable SLS integration directly on the group details page in the PAI console, configuring SLSProject and LogStore and letting the platform handle machine‑group creation and heartbeat monitoring.[^27] These logs can then be queried, visualized, and used to configure alerts for application‑level anomalies, scaling issues, and failed inferences.

### 5.3 Control‑plane audit logging

ActionTrail is the primary service for auditing control‑plane operations across Alibaba Cloud; it records console, API, and SDK actions as events for most services, including PAI and ACK.[^10][^28] By default, ActionTrail retains the last 90 days of events in the console, but you can create trails to deliver events to OSS or SLS logstores for long‑term retention, centralized search, and integration with downstream analytics or SIEM tooling.[^10][^12] Alibaba Cloud highlights that log retention is important for compliance frameworks such as China’s MLPS 2.0, which requires operation logs to be stored for at least 180 days, making ActionTrail plus SLS or OSS a key component for compliance in Lingjun environments.[^10][^11]

### 5.4 Centralized log audit and data activity monitoring

Log Audit Service, built on SLS, provides automated collection and centralized storage of audit logs from many services including ActionTrail, ACK, OSS, NAS, SLB, VPC, RDS, WAF, Anti‑DDoS, Cloud Firewall, and Security Center, addressing fragmentation of logs across accounts and services.[^11] This enables unified security auditing and threat detection for Lingjun environments by aggregating compute, network, storage, and security service logs into a single project with query and alerting capabilities.[^11][^29] Data Security Center provides fine‑grained audit logs for data assets such as RDS, PolarDB, OSS, and MaxCompute, storing them in SLS projects and logstores, which can be correlated with Lingjun workload logs to investigate data exfiltration or abnormal data access patterns.[^30]

## 6. Security best practices for Lingjun deployments

### 6.1 Network and infrastructure hardening

Deploy Lingjun and ACK Lingjun clusters into dedicated VPCs with well‑planned private CIDR ranges that avoid overlap with on‑premises or other cloud networks, as required by ACK Lingjun cluster creation.[^6] Use security groups and route tables to implement tiered segmentation (for example, management plane, training plane, inference endpoints) and restrict inbound Internet access to hardened ALB/SLB frontends that terminate TLS using SSL Certificate Service certificates.[^7][^17][^6] Where possible, integrate Cloud Firewall, WAF, and Anti‑DDoS with public‑facing components to provide additional network‑ and application‑layer protections for Lingjun workloads.[^17][^11]

### 6.2 Compute and platform security

Treat Lingjun nodes and ACK worker nodes as high‑value assets, applying hardened OS images, regular patching, and Security Center agents or equivalent host‑based protection where supported.[^17][^6] For containerized workloads on ACK Lingjun clusters, use Kubernetes best practices such as restrictive pod security (or equivalent), namespace‑based isolation, network policies, and limited hostPath usage, combined with RAM roles for service accounts where available.[^6][^18] Ensure that CI/CD pipelines that deploy models and training code into Lingjun clusters use secure artifact repositories and enforce code‑signing or integrity checks before deployment.

### 6.3 Data protection and key management

Encrypt all data at rest for Lingjun workloads by enabling disk encryption for ECS and CPFS volumes and server‑side encryption for OSS buckets and databases used as data sources or model registries.[^8][^9][^7] Use customer‑managed KMS keys for highly sensitive datasets and models, applying strict RAM policies to keys, enabling rotation, and monitoring KMS usage through logs streamed to SLS.[^7][^9][^11] Enforce TLS for all client‑to‑service and service‑to‑service communications (for example, HTTPS for inference endpoints, TLS for database connections), and consider application‑level encryption via KMS SDKs for especially sensitive payloads.[^7][^8]

### 6.4 IAM and access‑control hygiene

Implement a strong RAM strategy by avoiding daily use of the root account and instead creating RAM users and roles with least‑privilege access to Lingjun and PAI resources.[^20][^21] Enable MFA for the root account and all RAM users with console access, enforce strong password policies (length, character complexity, expiration, and reuse prevention), and regularly disable or remove inactive users and unused AccessKeys as recommended by RAM and best‑practice advisories.[^21][^22][^23] Use condition keys such as acs:SourceIp, pai:Accessibility, and pai:EntityAccessType to restrict access paths and emphasize creator‑only access for private resources, and avoid broad policies such as AliyunPAIFullAccess except for tightly controlled break‑glass roles.[^18][^5][^24]

### 6.5 Logging, monitoring, and incident response

Standardize on SLS as the central log platform and configure ActionTrail trails to deliver account‑ and service‑level audit events to a dedicated log project, with retention meeting or exceeding MLPS or internal policy requirements.[^10][^11][^12] Enable Log Audit Service to automatically onboard logs from ACK, VPC, OSS, SLB, WAF, Cloud Firewall, and Security Center so that Lingjun activities can be correlated across layers, and integrate SLS with CloudMonitor or external SIEM/XDR platforms for alerting and analytics.[^11][^29] For PAI and Lingjun workload logs, configure PAI resource‑group log collection to SLS and define alert rules for anomalous behavior (for example, repeated job failures, unexpected scaling, abnormal latencies), building runbooks that incorporate both SLS queries and ActionTrail investigation steps.[^27][^26]

## 7. Findings, risks, and recommendations

The following table summarizes key Lingjun security concerns, associated risks, and recommended guardrails.

### 7.1 Summary of key risks and guardrails

| Area | Concern | Risk | Recommendation | Proposed guardrail |
|------|---------|------|----------------|--------------------|
| Data at rest | Reliance on default service‑managed KMS keys for disks, OSS, and databases backing Lingjun workloads | Compromise of an Alibaba Cloud account or misconfigured RAM policy could expose large volumes of sensitive training data and models encrypted under shared default keys | Use customer‑managed KMS keys for critical datasets and models, with strict RAM policies, separation of duties for key admins, and regular rotation | Policy baseline requiring CMKs (not default keys) for specified OSS buckets, ECS disks, CPFS volumes, and databases tagged as `classification=high` or `classification=restricted`[^7][^8][^9] |
| Data in transit | Incomplete adoption of TLS for internal service‑to‑service traffic and database connections from Lingjun workloads | Attackers gaining a foothold in the VPC could sniff or tamper with plaintext traffic, including credentials and training data | Enforce HTTPS for all public endpoints, TLS for all DB and message‑bus connections, and consider mTLS for intra‑cluster services where supported | Organization‑wide requirement that new listeners use HTTPS only; periodic compliance scans on SLB/ALB and RDS instances; admission controls in ACK to block non‑TLS endpoints where possible[^7][^8][^6] |
| Network and infra | Flat VPC topology for Lingjun clusters with broad security‑group rules and direct Internet exposure | Expanded blast radius for lateral movement and DDoS/WAF evasion, with limited ability to contain compromised workloads | Place Lingjun and ACK Lingjun in dedicated VPCs, segment management/training/inference tiers, and expose only hardened ALB/SLB frontends with WAF and Anti‑DDoS | Baseline network architecture with mandatory separate subnets and security groups per tier; security‑group linting rules; guardrail templates for ACK Lingjun cluster creation (VPC, CIDR, SG) [^6][^17][^11] |
| IAM | Over‑privileged identities using AliyunPAIFullAccess or wildcard RAM policies for PAI/Lingjun administration | Misuse or compromise of admin credentials could lead to cluster deletion, data exfiltration, or injection of malicious training code | Adopt least‑privilege RAM policies for Lingjun, using resource‑level ARNs and conditions (pai:Accessibility, acs:SourceIp), and constrain AliyunPAIFullAccess to break‑glass roles with MFA | Periodic RAM policy audit for wildcard actions/resources; preventive checks (for example, OPA or custom scanners) blocking overly broad policies; mandatory MFA for any role with Lingjun admin rights[^20][^21][^18][^5][^24] |
| Credentials | Long‑lived AccessKeys on Lingjun‑managing automation or CI/CD systems | Stolen or leaked keys provide persistent access to PAI, ACK, and storage APIs involved in Lingjun workflows | Prefer RAM roles and STS‑issued temporary credentials for workloads and automation; rotate any necessary long‑lived keys regularly and monitor their usage | CIS‑style control requiring use of RAM roles for ECS/ACK workloads; automated checks for stale or unused AccessKeys; alerts on anomalous STS usage patterns[^21][^22][^23] |
| Logging and audit | Partial or inconsistent enablement of ActionTrail trails, Log Audit Service, and PAI/SLS log collection | Gaps in forensic visibility and compliance evidence, making incident investigation and MLPS/industry audits difficult | Enable multi‑account ActionTrail trails to SLS/OSS with ≥180‑day retention, onboard all critical services into Log Audit Service, and standardize Lingjun workload logging to SLS | Organization standard requiring ActionTrail trails and Log Audit Service in all accounts; periodic checks for log source coverage; default SLS projects/logstores for Lingjun clusters and workspaces[^10][^11][^12][^27] |
| Monitoring and response | Lack of actionable alerts on Lingjun workload anomalies and security signals from SLS | Slow detection of compromised jobs, data exfiltration, or disruptive configuration changes | Define alert rules in SLS and CloudMonitor for key metrics (job failures, unusual scaling, error spikes) and integrate SLS with a SOC/SIEM for correlation and playbooks | Central SLS project for security alerts; minimum baseline of SLS saved searches and alert rules; documented incident‑response runbooks referencing SLS and ActionTrail queries[^25][^26][^11][^29] |
| Data governance | No standardized classification of datasets and models used in Lingjun training and inference | Inconsistent application of encryption, access controls, and retention to high‑sensitivity assets | Implement data‑classification tags on storage and Lingjun resources and map them to required encryption, access, and logging controls | Guardrail policies that enforce CMK usage, stricter RAM policies, and longer log retention for resources tagged at higher classification levels, implemented via tagging standards and policy‑as‑code engines integrated with RAM/SLS[^7][^8][^30] |

### 7.2 Overall assessment

Lingjun’s security posture is largely determined by how tenants configure foundational Alibaba Cloud services—VPC, RAM, KMS, and SLS—rather than by service‑specific controls, so a strong shared‑controls program is essential for secure adoption.[^1][^2][^17] When combined with disciplined IAM, pervasive encryption, centralized logging, and mature incident‑response processes, Lingjun can meet demanding security and compliance requirements for large‑scale AI workloads.[^7][^9][^10][^11] Conversely, inadequate configuration of IAM, logging, and network segmentation can significantly increase the blast radius of breaches in Lingjun clusters due to their concentration of compute and data.[^17][^6][^11]
